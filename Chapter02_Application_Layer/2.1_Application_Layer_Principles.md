## ✅ 知识点1: 网络应用的体系结构

**理论**
* **网络应用的实现位置**：
  * 网络核心中没有应用层软件
  * 网络应用只在端系统上存在
  * 实现快速的应用开发和部署
* **可能的应用架构**：
  * **客户-服务器模式**（C/S: client/server）
  * **对等模式**（P2P: Peer To Peer）
  * **混合体**：客户-服务器和对等体系结构

---

<a id="id2"></a>
## ✅ 知识点2: 客户-服务器（C/S）体系结构

**服务器（Server）**
* 一直运行
* 固定的IP地址和周知的端口号（约定）
* 扩展性：服务器场（数据中心进行扩展）
* **缺点**：扩展性差

**客户端（Client）**
* 主动与服务器通信
* 与互联网有间歇性的连接
* 可能是动态IP地址
* 不直接与其它客户端通信

---

<a id="id3"></a>
## ✅ 知识点3: P2P体系结构

**特点**
* （几乎）没有一直运行的服务器
* 任意端系统之间可以进行通信
* 每一个节点既是客户端又是服务器
* **自扩展性**：新peer节点带来新的服务能力，当然也带来新的服务请求
* 参与的主机间歇性连接且可以改变IP地址

**例子**
* Gnutella
* 迅雷

---

<a id="id4"></a>
## ✅ 知识点4: C/S和P2P混合体系结构

**Napster模式**
* **文件搜索**：集中
  * 主机在中心服务器上注册其资源
  * 主机向中心服务器查询资源位置
* **文件传输**：P2P
  * 任意Peer节点之间

**即时通信**
* **在线检测**：集中
  * 当用户上线时，向中心服务器注册其IP地址
  * 用户与中心服务器联系，以找到其在线好友的位置
* **两个用户之间聊天**：P2P

<a id="id5"></a>
## ✅ 知识点5: 进程通信

**进程（Process）**
* 在主机上运行的应用程序

**通信方式**
* **同一个主机内**：使用进程间通信机制通信（操作系统定义）
* **不同主机**：通过交换报文（Message）来通信

**角色划分**
* **客户端进程**：发起通信的进程
* **服务器进程**：等待连接的进程

**注意**
* P2P架构的应用也有客户端进程和服务器进程之分

<a id="id6"></a>
## ✅ 知识点6: 分布式进程通信需要解决的问题

**问题1：进程标示和寻址问题**（服务用户）

**问题2：传输层-应用层提供服务是如何**（服务）
* 位置：层间界面的SAP（TCP/IP：socket）
* 形式：应用程序接口API（TCP/IP：socket API）

**问题3：如何使用传输层提供的服务实现应用**（使用服务，实现应用）
* 借助传输层提供的服务
* 定义应用层协议：报文格式，解释，时序等
* 编制程序，使用OS提供的API，调用网络基础设施提供通信服务传报文，实现应用时序等

<a id="id7"></a>
## ✅ 知识点7: 对进程进行编址(addressing)

**进程标识要素**
* **主机**：唯一的32位IP地址
  * 仅仅有IP地址不能够唯一标示一个进程；在一台端系统上有很多应用进程在运行
* **所采用的传输层协议**：TCP or UDP
* **端口号（Port Numbers）**
  * 一些知名端口号的例子：
    * HTTP: TCP 80
    * Mail: TCP 25
    * FTP: TCP 21

**端节点构成**
* 一个进程：用 **IP + port** 标示端节点
* 本质上，一对主机进程之间的通信由2个端节点构成

---

<a id="id8"></a>
## ✅ 知识点8: 传输层提供的服务——层间信息

**层间接口必须携带的信息**

* **谁传的**：对等的应用进程的标示：源IP + TCP(UDP)端口
* **传给谁**：对方的应用进程的标示：对方的IP + TCP(UDP)端口号

**传输层实体的工作**
* 根据这些信息进行TCP报文段（UDP数据报）的封装
  * 源端口号，目标端口号，数据等
* 将IP地址往下交IP实体，用于封装IP数据报：源IP，目标IP

**Socket API的简化**
* 如果每次传输报文都携带如此多的信息，太繁琐易错，不便于管理
* 用个**代号**标示通信的双方或者单方：**socket**
  * 就像OS打开文件返回的句柄一样
  * 对句柄的操作，就是对文件的操作

---

<a id="id9"></a>
## ✅ 知识点9: TCP之上的套接字(socket)

**TCP Socket的定义**
* 对于使用面向连接服务（TCP）的应用而言，套接字是**4元组**的一个具有本地意义的标示
* **4元组**：(源IP，源port，目标IP，目标port)
  * 唯一的指定了一个会话（2个进程之间的会话关系）

**TCP Socket的特点**
* TCP服务，两个进程之间的通信需要之前要建立连接
  * 两个进程通信会持续一段时间，通信关系稳定
* 可以用一个**整数**表示两个应用实体之间的通信关系（本地标示）
* 穿过层间接口的信息量最小
* 应用使用这个标示与远程的应用进程通信，不必在每一个报文的发送都要指定这4元组
* 就像使用操作系统打开一个文件，OS返回一个文件句柄一样，以后使用这个文件句柄，而不是使用这个文件的目录名、文件名

**TCP Socket内容**
* 源IP、源端口、目标IP、目标端口、状态等

---

<a id="id10"></a>
## ✅ 知识点10: UDP之上的套接字(socket)

**UDP Socket的定义**
* 对于使用无连接服务（UDP）的应用而言，套接字是**2元组**的一个具有本地意义的标示
* **2元组**：IP，port（源端指定）
  * UDP套接字指定了应用所在的一个端节点（end point）

**UDP Socket的特点**
* UDP服务，两个进程之间的通信无需建立连接
  * 每个报文都是独立传输的
  * 前后报文可能给不同的分布式进程
* 只能用一个整数表示本应用实体的标示
* 穿过层间接口的信息大小最小

**发送/接收报文时的要求**
* **发送报文时**：
  * 采用创建好的本地套接字（标示ID），就不必在发送每个报文中指明自己所采用的IP和port
  * 但必须要指定对方的IP和UDP port（另外一个端节点）
* **接收报文时**：
  * 传输层需要上传对方的IP，port

---

## 套接字（Socket）总结

**核心比喻**
* 套接字 <-> **门户**
* 发送进程将报文推出门户，发送进程依赖于传输层设施在另外一侧的门将报文交付给接受进程
* 接收进程从另外一端的门户收到报文（依赖于传输层设施）

**控制归属**
| 层级 | 控制者 |
|------|--------|
| 应用层、socket、应用进程 | 应用开发者控制 |
| 传输层及以下 | 操作系统控制 |

<a id="id11"></a>
## ✅ 知识点11: 应用层协议

**定义**
* 定义了：运行在不同端系统上的应用进程如何相互交换报文

**内容要素**
* 交换的报文类型：请求和应答报文
* 各种报文类型的语法：报文中的各个字段及其描述
* 字段的语义：即字段取值的含义
* 进程何时、如何发送报文及对报文进行响应的规则

**协议类型**

| 类型 | 特点 | 例子 |
|------|------|------|
| **公开协议** | 由RFC文档定义，允许互操作 | HTTP, SMTP |
| **专用（私有）协议** | 协议不公开 | Skype |

**应用协议的实现**
* 定义应用层协议：报文格式，解释，时序等
* 编制程序，通过API调用网络基础设施提供通信服务传报文，解析报文，实现应用时序等
* 应用协议仅仅是应用的一个组成部分
  * 例：Web应用 = HTTP协议 + web客户端 + web服务器 + HTML

<a id="id12"></a>
## ✅ 知识点12: 常见应用对传输服务的要求

**描述维度**

| 维度 | 说明 |
|------|------|
| **数据丢失率** | 有些应用要求100%可靠数据传输；有些能容忍一定比例的数据丢失 |
| **吞吐** | 一些应用需要最小限度的吞吐；一些能弹性利用可用吞吐 |
| **时间敏感性** | 一些应用对数据传输有严格的时间限制（延迟、延迟差） |
| **安全性** | 机密性、完整性、可认证性（鉴别） |

**常见应用需求表**

| 应用 | 数据丢失率 | 吞吐 | 时间敏感性 |
|------|-----------|------|-----------|
| 文件传输 | 不能丢失 | 弹性 | 不敏感 |
| e-mail | 不能丢失 | 弹性 | 不敏感 |
| Web文档 | 不能丢失 | 弹性 | 不敏感 |
| 实时音视频 | 容忍丢失 | 音频：5kbps-1Mbps<br>视频：0kbps-5Mbps | 是，100ms |
| 存储音视频 | 容忍丢失 | 同上 | 是，几秒 |
| 交互式游戏 | 容忍丢失 | 几kbps ~ 10kbps | 是，100ms |
| 即时讯息 | 不能丢失 | 弹性 | 是和不是 |

## ✅ 知识点13: Internet传输层提供的服务

### TCP服务

| 特性 | 说明 |
|------|------|
| **可靠传输** | 保证数据可靠到达 |
| **流量控制** | 发送方不会淹没接受方 |
| **拥塞控制** | 当网络出现拥塞时，能抑制发送方 |
| **面向连接** | 要求在客户端进程和服务器进程之间建立连接 |
| **不提供** | 最小吞吐保证、时间保证、安全性 |

### UDP服务

| 特性 | 说明 |
|------|------|
| **不可靠数据传输** | 不保证到达 |
| **无连接** | 无需建立连接 |
| **无拥塞控制** | 应用能够按照设定的速度发送数据 |
| **不提供** | 可靠性、流量控制、拥塞控制、时间/带宽保证、安全性 |

### UDP存在的必要性

* **区分进程**：能够区分不同的进程，而IP服务不能
  * 在IP提供的主机到主机端到端功能的基础上，区分了主机的应用进程
* **无需建立连接**：省去了建立连接时间，适合事务性的应用
* **不做可靠性工作**：例如检错重发，适合那些对实时性要求比较高而对正确性要求不高的应用
  * 为了实现可靠性（准确性、保序等），必须付出时间代价（检错重发）
* **无流量/拥塞控制**：应用能够按照设定的速度发送数据
  * 在TCP上面的应用，应用发送数据的速度和主机向网络发送的实际速度是不一致的

### Internet应用及其协议

| 应用 | 应用层协议 | 传输协议 |
|------|-----------|----------|
| e-mail | SMTP [RFC 2821] | TCP |
| 远程终端访问 | Telnet [RFC 854] | TCP |
| Web | HTTP [RFC 2616] | TCP |
| 文件传输 | FTP [RFC 959] | TCP |
| 流媒体 | 专用协议（如RealNetworks） | TCP或UDP |
| Internet电话 | 专用协议（如Net2Phone） | TCP或UDP |

### 安全TCP（SSL）

**问题**
* TCP和UDP都没有加密
* 明文通过互联网传输，甚至密码

**SSL解决方案**
* SSL在应用层，下层的传输协议还是TCP
* 应用采用SSL库，SSL库使用TCP通信
* 应用通过API将明文交给socket，SSL将其加密在互联网上传输
* 提供：
  * 私密性
  * 数据完整性
  * 端到端的鉴别

## 🔑 核心要点总结

1. **三种体系结构**：C/S（有中心服务器）、P2P（无中心、自扩展）、混合（搜索集中+传输P2P）
2. **进程通信**：同一主机用OS机制，不同主机交换报文
3. **进程编址**：IP地址 + 端口号（Port）唯一标识进程
4. **Socket本质**：层间接口的本地代号
   * TCP Socket = 4元组（源IP、源Port、目标IP、目标Port）
   * UDP Socket = 2元组（本IP、本Port）
5. **TCP vs UDP**：
   * TCP：可靠、流量控制、拥塞控制、面向连接
   * UDP：无连接、无拥塞控制、低开销、适合实时应用
6. **应用层协议**：定义报文类型、语法、语义、时序；分公开协议和私有协议

## 📌 考试速记版

**Socket对比**

| | TCP Socket | UDP Socket |
|--|-----------|-----------|
| 维度 | 4元组 | 2元组 |
| 连接 | 面向连接（先建立连接） | 无连接 |
| 标示范围 | 会话（一对进程） | 端节点（单个进程） |
| 发送报文 | 只需socket ID | 需socket ID + 对方IP/Port |

**TCP vs UDP 口诀**
> "TCP三控一可：可靠、流控、拥控、面向连接；
> UDP三无两不：无连接、无可靠、无拥控，不保证、不等待"

**知名端口号**
* HTTP: 80
* FTP: 21
* SMTP: 25
