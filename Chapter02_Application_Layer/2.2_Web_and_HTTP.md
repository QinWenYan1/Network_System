## ✅ 知识点1: Web与HTTP的一些术语

**理论**
* **Web页(Web Page)**：由一些<b>对象(Object)</b>组成
  * **对象**：可以是HTML文件、JPEG图像、Java小程序、声音剪辑文件等
* **Web页结构**：
  * 含有一个**基本的HTML文件**，该基本HTML文件包含若干**对象的引用（链接）**
  * 该链接是通过URL格式标识的
  * 因此可以通过URL找到对象并画到web页面上
* **URL引用**：通过URL对每个对象进行引用和访问
  * 访问协议、用户名、口令字、端口等

<a id="id2"></a>
## ✅ 知识点2: URL格式

**理论**
* **URL格式**：
`Prot://user:psw@www.someSchool.edu/someDept/pic.gif:port`

| 组成部分 | 示例 | 说明 |
|----------|------|------|
| **协议名(Prot)** | `http://` | 访问协议 |
| **用户:口令(user:psw)** | `user:psw@` | 认证信息 |
| **主机名** | `www.someSchool.edu` | 服务器域名 |
| **路径名** | `/someDept/pic.gif` | 资源路径 |
| **端口(:port)** | `:80` | 服务端口号 |

**注意点**
* 📋 **术语**：URL (Uniform Resource Locator) - 统一资源定位符

<a id="id3"></a>
## ✅ 知识点3: HTTP概况

**理论**
* **HTTP**：**超文本传输协议(HyperText Transfer Protocol)**
  * Web的**应用层协议**
* **客户/服务器（C/S）模式**
  * **客户**：请求、接收和显示Web对象的**浏览器**
  * **服务器**：对请求进行响应，发送对象的**Web服务器**
* **版本**
  * **HTTP 1.0**：RFC 1945
  * **HTTP 1.1**：RFC 2068

---

<a id="id4"></a>
## ✅ 知识点4: HTTP使用TCP

**理论**
1. **客户发起**一个与服务器的**TCP连接**（建立套接字），**端口号为80**
2. **服务器接受**客户的TCP连接，并建立这个会话关系的socket
3. 在**浏览器(HTTP客户端)**与**Web服务器**(HTTP服务器)之间**交换HTTP报文**（应用层协议报文）
4. web浏览器拿到如果是html的文件就解释出来根据URL 
5. html文件解释完后**TCP连接关闭**

---

<a id="id5"></a>
## ✅ 知识点5: HTTP是无状态的

**理论**
* **服务器并不维护关于客户的任何信息**：响应完成后就完事了
* **维护状态的协议很复杂！**
  * 必须维护历史信息（状态）
  * 如果服务器/客户端死机，它们的状态信息可能不一致，二者的信息必须一致
* **无状态的服务器能够支持更多的客户端**

**注意点**
* ⚠️ **关键概念**：HTTP是**无状态协议(Stateless Protocol)**，服务器不保存客户状态

<a id="id6"></a>
## ✅ 知识点6: HTTP连接类型概述

**理论**

| 类型 | 说明 |
|------|------|
| **非持久HTTP** | 最多只有一个对象在TCP连接上发送，下载多个对象需要多个TCP连接，HTTP/1.0使用 |
| **持久HTTP** | 多个对象可以在一个（客户端和服务器之间的）TCP连接上传输，HTTP/1.1默认使用 |

<a id="id7"></a>
## ✅ 知识点7: 非持久HTTP连接

**理论**
假设用户输入URL：`www.someSchool.edu/someDept/home.index`

1. **HTTP客户端**在端口号80发起一个到服务器`www.someSchool.edu`的**TCP连接**
2. **HTTP服务器**在80号端口等待连接，接受连接并通知客户端
3. **HTTP客户端**向TCP连接的套接字发送**HTTP请求报文**，报文表示客户端需要对象`someDepartment/home.index`
4. **HTTP服务器**接收到请求报文，检索出被请求的对象，将对象封装在一个**响应报文**，并通过其套接字向客户端发送
5. **TCP连接关闭**

<a id="id8"></a>
## ✅ 知识点8: 响应时间模型

**理论**
* **文件传输时间 = 2RTT + 文件传输时间**
* **RTT (Round-Trip Time)**：往返时间，指一个短分组从客户到服务器然后再返回客户所花费的时间
* **过程**
  * **1个RTT**：发起TCP连接
  * **1个RTT**：HTTP请求和等待响应
  * **文件传输时间**：传输的对象大小无法忽略不计

<a id="id9"></a>
## ✅ 知识点9: 持久HTTP连接

**非持久http的缺点**:
* 每个对象要**2个RTT**
* OS需为**每个TCP连接分配资源**
* 浏览器通常开**并行TCP连接**获取引用对象

**持久连接定义**：服务器发送响应后**保持TCP连接**
* 在**同一对客户端-服务器之间**的后续请求和响应报文通过该连接发送
* 持久HTTP连接方式又有两种：1. 流水线 2.非流水线

---

<a id="id10"></a>
## ✅ 知识点10: 持久HTTP非流水方式

**非流水方式(Non-Pipelined)**
  * 客户端**收到前一个响应后**才能发送新请求
  * **每个引用对象花费1个RTT**

---

<a id="id11"></a>
## ✅ 知识点11: 持久HTTP流水方式

**理论**
* **流水方式(Pipelined)** - HTTP/1.1默认使用
  * 客户端连续发送多个请求，无需等待响应
  * 客户端在收到前一个响应之前，可以连续发送后续的请求报文
  * **所有引用对象只花费1个RTT**

**注意点**
* 💡 流水方式是HTTP/1.1的默认方式，效率更高

<a id="id12"></a>
## ✅ 知识点12: HTTP请求报文格式

**理论**
* **HTTP请求报文示例**

```
GET /somedir/page.html HTTP/1.1
Host: www.someschool.edu
User-agent: Mozilla/5.0
Connection: close
Accept-language: cn

```

**结构说明**
* **请求行**：方法(GET/POST/HEAD) + URL + HTTP版本
* **首部行**：Host, Connection, User-agent, Accept-language等
* **换行回车符号**：表示报文结束
* **采用格式**: ASCII(人能阅读的)
* 两种类型的HTTP报文：请求，响应

## ✅ 知识点15: 提交表单输入方式

**理论**
* **POST方式**
  * 网页通常包含**表单输入**
  * 包含在<b>实体主体(entity body)</b>中发送到服务器
  * 例子：登录密码输入框用 POST
* **URL方式（使用GET方法）**
  * 输入通过**请求行的URL字段**上载
  * 浏览器把你的填写内容全部塞进网址地址栏（变成 `?名字=张三&年龄=20` 这种格式），然后带着这个长网址去问服务器要结果。
  * 所以 GET 确实主要用于"请求资源"，但提交表单数据本身就是一种请求资源的行为
  * 例子：你日常看到的搜索框用 GET

<a id="id16"></a>
## ✅ 知识点16: HTTP响应报文格式

**理论**
* **HTTP响应报文示例**

```
HTTP/1.1 200 OK
Connection: close
Date: Thu, 06 Aug 1998 12:00:15 GMT
Server: Apache/1.3.0 (Unix)
Last-Modified: Mon, 22 Jun 1998 09:23:24 GMT
Content-Length: 6821
Content-Type: text/html

data data data...
```

**结构说明**
* **状态行**：协议版本 + 状态码 + 状态信息
* **首部行**：Connection, Date, Server, Last-Modified（相当于版本号）, Content-Length（提供了边界，http才知道边界区分去维护，因为tcp不会去维护）, Content-Type
* **数据**：请求的HTML文件等

<a id="id17"></a>
## ✅ 知识点17: HTTP响应状态码

**理论**

| 状态码 | 含义 | 说明 |
|--------|------|------|
| **200 OK** | 请求成功 | 请求对象在响应报文中 |
| **301 Moved Permanently** | 永久转移 | 新URL在Location首部行中指定 |
| **400 Bad Request** | 请求错误 | 服务器无法解读请求 |
| **404 Not Found** | 未找到 | 请求的文档不在服务器上 |
| **505 HTTP Version Not Supported** | 版本不支持 | 服务器不支持该HTTP版本 |

**注意点**
* 📋 200系列-成功、300系列-重定向、400系列-客户端错误、500系列-服务器错误

## ✅ 知识点18: 用户-服务器状态-Cookies

**理论**
* **无状态协议的问题**：服务器无法维护与客户端会话状态
  * 如购物网站需要维护客户端状态来为客户推荐物品
* **Cookies用途**：维持HTTP无状态协议之上的**用户-服务器状态**

<a id="id21"></a>
## ✅ 知识点21: Web缓存(代理服务器)

**理论**
* **目标**：不访问**原始服务器**，就满足客户的请求
* **用户配置**：用户设置浏览器**通过缓存访问Web**
* **缓存行为**
  * 浏览器将**所有HTTP请求发给缓存**，当第一次请求发给原始服务器的时候
  * HTTP响应也发给缓存和浏览器让缓存代理获得资源
  * 缓存**既充当客户端，也充当服务器**
  * 下一次客户端浏览器可以直接从代理服务器获得资源
  * 通常由**ISP安装**（大学、公司、居民区ISP）

<a id="id22"></a>
## ✅ 知识点22: 为什么使用Web缓存

**理论**
* **原因1：降低响应时间**
  * 缓存离客户端更近
* **原因2：减少机构内部网络与Internet接入链路上的流量**
* **原因3：使较弱ICP也能有效提供内容**
  * 通过缓存大量内容，减少对原始服务器带宽需求

---

<a id="id23"></a>
## ✅ 知识点23: 条件GET方法

**理论**
* **问题**：original server的资源更新了但是代理服务器的却没有更新
* **目标**：如果缓存器中的对象拷贝是最新的，**就不要发送对象**
1. 当用户请求代理的资源的时候，缓存器向源服务器发送一个请求
2. **缓存器在HTTP请求报文中添加一个头部指定缓存拷贝的日期**：`If-modified-since: <date>`
3. **服务器响应**
    * **对象未修改**：发送一个头部response告知没有改动 - `HTTP/1.0 304 Not Modified`，**不发送对象**
    * **对象已修改**：`HTTP/1.0 200 OK` + 发送新对象到代理服务器

<a id="id24"></a>
## ✅ 知识点24: 缓存示例分析

**理论**
* **场景假设**
  * 平均对象大小 = 100KB
  * 平均请求率 = 15请求/s
  * 平均到浏览器速率 = 1.5Mbps
  * Internet延时 = 2s
  * 接入链路带宽 = 1.54Mbps

* **方案对比**

| 方案 | 接入链路带宽 | 利用率 | 总延迟 |
|------|-------------|--------|--------|
| 原始 | 1.54Mbps | 99% | 几分钟 |
| 升级链路 | 154Mbps | 9.9% | ~2s（昂贵）|
| 本地缓存(命中率0.4) | 1.54Mbps | 58% | ~1.2s（便宜）|

**结论**：缓存方案**成本更低，效果更好**
* 用户通过proxy server访问，如果有缓存就直接访问，如果没有proxy server先去缓存了original server的资源后，再分发给本地用户
  * 如果用户有40%的概率能从缓存获得资源的话：
    * 总体延迟 = 0.6 * (从原始服务器获取对象的延迟) + 0.4 * (从缓存获取对象的延迟)

## 🔑 核心要点总结
1. **HTTP是应用层协议**，使用TCP端口80，是无状态协议
2. **两种连接类型**：非持久（HTTP/1.0，每个对象2个RTT）vs 持久（HTTP/1.1默认，流水方式1个RTT）
3. **请求/响应报文**：请求行+首部行+主体 / 状态行+首部行+数据
4. **状态码**：200成功、301重定向、404未找到、505版本不支持
5. **Cookie**：维持用户-服务器状态，但有隐私问题
6. **Web缓存**：减少延迟和流量，使用条件GET验证 freshness

## 📌 考试速记版
* **HTTP核心**：TCP/80端口、无状态、请求-响应模式
* **连接对比**：非持久(2RTT/对象) vs 持久流水(1RTT所有对象)
* **方法**：GET(获取)、POST(提交)、HEAD(头部)、PUT(上传)、DELETE(删除)
* **状态码**：2xx成功、3xx重定向、4xx客户端错、5xx服务器错
* **Cookie四组件**：Set-cookie首部、Cookie首部、客户端文件、后端数据库
* **缓存优势**：降低延迟、减少流量、条件GET(304 Not Modified)

**记忆口诀**：HTTP八零无状态，持久非持久两种连，Get请求Post提交，Cookie四件记心间，缓存命中省带宽
